# results/ — Raw Experiment Records (#21)

Every scenario's **raw numbers** are persisted here as versioned JSON, so all tables and
figures regenerate from data without re-running any model (NFR1). Each file is one run,
written by `ResultRecorder` (`src/airllm_bench/shared/recorder.py`).

## Record schema (PLAN §6)

```json
{
  "schema_version": "1.00",
  "code_version": "1.00",
  "timestamp": "<UTC ISO-8601>",
  "scenario": "<name>",
  "metrics": { "peak_ram_gb": ..., "ttft_s": ..., "throughput_tok_s": ..., ... },
  "...": "scenario-specific fields (model, engine, output, finding, ...)"
}
```

Filenames are `"<scenario>_<timestamp>.json"`; the analysis loader
(`src/airllm_bench/analysis/loader.py`) reads them and `latest_by_scenario` selects the
newest record per scenario.

## Notes on duplicate records (kept on purpose)

- **`airllm_7b_fp16` (2 records):** the first (12:02) is the **failed** RoPE attempt on
  transformers 4.48; the second (12:18) is the **success** on 4.42. Both kept as evidence of
  the dependency wall (see `reports/phase3_airllm_findings.md`).
- **`baseline_ollama_q4` (2 records):** the first (09:10) predates the SDK fix that persists
  `output`; the second (13:02) includes the generated text. `latest_by_scenario` uses the
  newer one.

## Reproducibility

The full set regenerates with the commands above. The two long runs
(`baseline_hf_direct.py` ≈ 2 h, `airllm_7b.py` ≈ 13 min, reusing shards) need the `ml` extra
(`uv run --extra ml python ...`); the analysis scripts need only the base env.
